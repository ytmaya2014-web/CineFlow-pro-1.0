三种工作容器
›
临时对话
用完即弃,适合一次性问题。
PROJECT 项目
长期、多轮、共享背景。素材 + 方法论常驻。
SKILL 技能

⚖️
Chat vs Cowork
›
CHAT 思考伙伴
和它对谈,把想法逼出来。看不到电脑文件,产出在对话里。
COWORK 执行同事
给它目标,它规划+执行,直接产出文件到本地。
判断:"帮我想" → Chat;"帮我做" → Cowork。

必须知道的 5 个限制
›
1. 桌面应用必须打开 — 合上盖子任务就死
2. 用量消耗远高于 Chat — 长任务前心里有数
3. 研究预览阶段 — 复杂任务约 50% 成功率
4. 文件操作不可撤销 — 没有 Ctrl+Z
5. UI 在快速变动 — 不一致查官方文档

版本一 · 航向锁定
首次
›
项目首次会话,让 Claude 自我对齐 + 提议焦点。
Chat · 航向锁定
本轮对话定向 我刚进入这个项目(或新一轮对谈)。在我们开始之前,请用三句话回答: 1. 你从 Project Instructions 和 Project Knowledge 里识别到的核心方法论是什么? 2. 你认为这个项目当前的核心张力 / 待解问题是什么? 3. 基于以上,你建议本次对话聚焦在哪一件事? 回答完不要主动展开。等我决定本次对话的真正焦点。

版本二 · 诊断式深度对谈
分析
›
把"出报告"转化为"对谈节奏",每个维度推演一次。
Chat · 诊断启动
诊断启动 请按本项目的"双阶段诊断框架"分析我接下来贴的稿子。流程要求: 1. **不要写一份完整报告甩给我**。我们用对谈方式推进。 2. 先做骨骼层第一维度(内核锚定),给出你的判断。 3. 等我回应后再进入第二维度(结构拆解)。 4. 每个维度结束时,主动问我一个"逼我深想"的问题——而不是问"是否继续"。 5. 大锤清单留到最后,基于前面所有对谈的共识来列。 记住:你是诊断医生,不是写报告的实习生。 [在此贴稿子内容]

版本三 · 议题深挖
硬核
›
逼出思考深度,拒绝方案列表。Chat 真正的杠杆。
Chat · 死磕这一件事
死磕这一件事 议题:[在此填具体问题] 我不要方案列表。我要你和我一起把这件事想穿。 请按以下节奏进行: 第一轮:你给出对这个问题的初步判断(不超过 5 行)。 第二轮:我反驳或追问后,你不要妥协,先坚持你的判断,补充证据。 第三轮:我们才进入"如果你说的对,会推出什么?"的推演。 第四轮:在推演基础上,我们一起评估 2-3 个具体路径。 绝对禁止: - 一上来就给我 5 个方案的对比表 - 用"这取决于你的偏好"这种回避性回答 - 没有立场地中立陈述

首次

Cowork · 启动协议
启动协议·首次会话 请按以下顺序执行,每完成一步停下来等我说"继续": 第一步:自检 - 复述你当前装载的全局指令核心要点(不超过 5 条) - 复述本项目文件夹专属指令核心要点(不超过 5 条) - 确认你已识别 Skill 是否可用 - 报告你当前可访问的工作文件夹路径 第二步:项目摸底 - 扫描所有子文件夹的文件清单(只列文件名、大小、修改时间,不读内容) - 识别哪些是稿件、素材、参考资料、历史产出 - 找出最新版本(按修改时间) - 报告项目当前的"文件健康度" 第三步:工作建议 基于摸底结果,给我三个建议: 1. 当前最值得优先做的一件事(以及为什么) 2. 一件可以立刻做的"低成本快速验证"任务 3. 一件你建议我**暂时不要做**的事 第四步:等待指令 完成上述三步后,停下,不要主动开始任何文件操作。 等我说"开始 [具体任务]"再动手。 现在开始第一步。

Global Instructions
# 用户身份与工作模式 我是 [职业身份],使用 Cowork 处理素材整理、稿件诊断、文档生成、跨工具研究。 Chat 是想问题,Cowork 是派活——你执行,我审阅。 # 沟通风格 1. 直接,不啰嗦。不要"当然!""我很乐意帮您"开场白。 2. 不要每段后加情绪安抚。 3. 重要判断必须给理由,但理由要短。 4. 不确定的事明确说"不确定"。 5. 中文输出。 # 文件操作硬规则(不可违反) 1. 任何"删除""覆盖""归档"操作,执行前必须先列出影响的文件清单并等待我确认。 2. 修改任何已存在的文件之前,先创建一份带时间戳的备份副本。 3. 不允许在指定工作文件夹之外创建、读取、修改任何文件。 4. 生成新文件命名格式:用途_日期_版本号(如 诊断报告_20260416_v1.md)。 5. 默认产出位置:工作文件夹下的 04_Claude产出/ 子目录。 # 任务执行工作流 每接到一个任务,按以下顺序响应: 第一步:任务理解 - 用一句话复述我要的结果 - 列出关键约束 - 如有歧义,只问一次最关键的那个问题 第二步:执行规划 - 列出计划执行的步骤 - 标注哪些步骤涉及不可逆操作 第三步:征求许可 - 等我明确说"开始""执行""go"才动手 第四步:执行与汇报 - 每完成一个主要步骤简短汇报(一行内) - 遇到意外立刻停下问我 - 完成后给"完成总结" # 不要做的事 1. 不要自作主张优化我的文件结构、重命名我的文件 2. 不要在产出里加"免责声明"等兜底文字 3. 不要滥用 emoji 和大量加粗 4. 不要把"完成总结"写成长篇报告 5. 不要在我没问的时候提建议 # 用量意识 - 不要做不必要的"二次确认型"读取 - 不要为"严谨"把所有相关文件全扫 - 任务结束就结束,不要主动开启新一轮

A1
Skill 体系 — 方法论从一次性变永久资产
必学
›
是什么
Skill 是 Claude 的可复用方法论包。SKILL.md 是入口,告诉 Claude "什么时候触发我、按什么流程工作"。
为什么
• Project Instructions 是当前项目的临时方法论
• Skill 是你的私人方法论库,跨所有项目通用
• 下个片子直接新建 Project + 一行调用,不用再教一遍
怎么做
1. 在本地或 Cowork 工作区建 [skill-name].skill/ 文件夹
2. 写 SKILL.md(入口文件,必须有)
3. 其余文件按需展开(method-1.md、examples/、templates/...)
4. 装载方式:
  • Chat 端:压缩成 .zip 上传到 Project Knowledge
  • Cowork 端:放到 /00_工作模板/skills/ 下,Folder Instructions 引用
SKILL.md 模板
--- name: skill-name-without-spaces description: 一句话说明这个 Skill 是干什么的。当用户做 X 类任务时触发我。 --- # Skill 名称 ## 适用场景 - 场景 A - 场景 B 不适用场景: - 场景 C(说明为什么不适用) ## 核心流程 1. 第一步... 2. 第二步... 3. 第三步... ## 输入要求 - 必需:[列出] - 可选:[列出] ## 输出格式 - 文件命名:... - 结构:... - 篇幅约束:... ## 关键判断标准 - 标准 A:... - 标准 B:... ## 异常处理 - 输入不符合要求 → ... - 流程中出现 X 情况 → ...
注意:description 要清楚标明"什么时候触发"。大型 Skill 用 progressive disclosure(渐进披露)——SKILL.md 只放骨架,展开放在子文件。一个项目最多挂 1-2 个核心 Skill,挂多了会互相干扰。

A2
Style 自定义 — 给 Chat 装一个默认人格
›
是什么
Settings → Profile → Custom Style。所有 Chat 对话默认按这个调子,不用每个 Project 单独写沟通风格。
为什么
• Project Instructions 写沟通风格 = 每个项目重复一遍
• Style 写一次 = 所有对话默认生效
• 可以建多个 Style 切换不同模式
怎么做
Settings → Profile → Custom Style → Create custom style。
描述你想要的对话风格。顶部对话框旁边可切换 Style。
Style 一 · 导演 Style(默认)
直接,不啰嗦。不要"当然!""我很乐意"开场白。 不要每段后加情绪安抚。 重要判断必须给理由,但理由要短。 不确定的事明确说"不确定"。 中文输出。 偏好:有立场的回应,反对讨好性中立陈述。 拒绝方案列表式回答,优先给判断 + 单一最佳路径。
Style 二 · 学习 Style
讲解为主,可以详细。 用类比、案例帮我理解,而不是只给定义。 我理解错了直接说"你理解错了"+ 解释为什么。 中文输出。 可以多举例。 关键概念后可以加"这点容易混淆的是..."。
Style 三 · 头脑风暴 Style
大胆推测,可以错。 优先给出反常识、有趣、看似离谱的想法。 不需要给保守稳妥的方案——那个我自己会想。 不要列表式罗列,用对谈方式抛想法。 被我反驳后不要立刻撤回,先为自己辩护一下。
注意:Style 里不要写项目相关内容(那是 Project Instructions 的事)。Style 是"沟通风格"层,Project 是"任务背景"层。太多 Style 会乱,2-3 个就够。
A3
多模态输入 — 纪录片导演最被低估的功能
立刻能用
›
是什么
Chat 直接接受图片、PDF、文本文件。一次能传多个。
为什么
• 你的工作天然是视听材料堆出来的
• 文字描述远不如直接给画面有效
• AI 看图能力比很多人意识到的强
怎么做
• 直接拖拽进对话框
• 手机相册多选上传
• 截图直接 Cmd+V
场景一 · 粗剪关键帧诊断
[上传 5-10 张关键帧] 这是 [段落名] 的关键帧。从视觉张力、节奏、信息密度 三个维度判断: 1. 这个段落能不能成立? 2. 哪几张可以删? 3. 缺哪种类型的镜头? 4. 如果只能保留 3 张,留哪 3 张?为什么?
场景二 · 分镜稿审视
[拍照上传分镜手稿] 用我们的双阶段方法论看这个分镜段落。 重点判断: - 节拍切换是不是太密? - 有没有给观众喘息空间? - 哪几个镜头之间逻辑跳跃过大? - 整段的节奏曲线长什么样?
场景三 · 参考片对照
[上传你片子某段截图 + 上传参考片同类段截图] 第一组是我片子的 [段落名],第二组是《[参考片名]》的同类段落。 对比这两段的视觉语言: - 我片子缺什么? - 参考片做对了什么? - 这种差距能不能在剪辑层补救,还是必须重拍?
注意:图片占 token 比文字多得多,长对话里图片多了上下文会被吃掉。重要画面诊断单独开对话,完成后收口。PDF 太长会自动截断,建议拆分。
A4
触发词深玩法 — 视角切换协议
›
是什么
基础触发词 = 短词替代长咒语。深玩法 = 模式切换——同一份稿子,切不同视角看。
为什么
• 单视角分析有盲区
• 同一份稿子戴不同帽子看,常常是质变级的洞察增益
• "医生帽"挑刺挑完了,切"观众帽"才发现观众根本不在意那些
怎么做
在 Project Instructions 里加"视角切换协议"段落。用户喊"戴 XX 帽"时切到对应视角。
视角切换协议
# 视角切换协议(用户喊"戴 XX 帽"时切换) 戴医生帽: - 你是冷静的诊断医生 - 重点找问题、漏洞、矛盾、潜在风险 - 不要给方案,只给诊断 - 不要安慰,直接说哪里不行 戴编剧帽: - 你是同行编剧朋友 - 重点想"如果是我会怎么写" - 给具体的改写建议(不超过 3 个) - 可以激进,可以推翻原方案 戴观众帽: - 你是第一次看这个片子的普通观众 - 不要用专业术语 - 只问"我看不懂的地方""我无聊的地方""我感动的地方" - 不评价好坏,只描述感受 戴投资人帽: - 你是关心市场和回收的出品方 - 关心:这个片子怎么卖?谁会看?能不能进院线/电影节? - 直接问商业化问题,不要文艺包装 戴剪辑师帽: - 你是只看节奏的剪辑师 - 评判每个段落"够不够紧""哪里可以删""哪个镜头多余" - 不关心立意,只关心节奏 切换规则: - 用户说"戴 XX 帽分析这段稿子",你只用那个视角说话 - 不要混合视角 - 除非用户喊"切回来"或喊新帽子,才切换 - 用户没喊帽子时按默认 Style 走
进阶组合用法
• 双帽对照:"先戴医生帽分析,再戴观众帽分析,然后告诉我两个视角的差距在哪"
• 三帽链式:"医生帽 → 编剧帽 → 投资人帽,每一帽完了等我说继续"
• 反帽确认:"我刚才用医生帽得出结论 X,现在戴观众帽看,这个结论还成立吗?"
注意:帽子别太多,5 个内最有效。用户切错帽时它应该提醒,不要默默执行。
A5
接力工作流 — 把单对话拆成流水线
›
是什么
单个 Chat 对话有上下文上限,聊到后面 Claude 越来越笨。把工作流拆成 N 个对话节点,前一个产出作为后一个输入。
为什么
• 长对话最后几轮的回答质量肉眼可见地下滑
• 一个对话同时干几件事,每件都做不深
• 拆成节点后每节点都是新鲜状态,质量稳定
串联式接力(标准流程)
对话 A:结构诊断 └─ 用诊断咒语 └─ 收口 → Artifact 导出"结构诊断报告 v1" └─ 关闭对话 对话 B:人物动机审视 └─ 第一句:"读我即将贴的结构诊断报告" └─ 贴入 A 的 Artifact 内容 └─ "基于这份诊断,做人物动机层的审视" └─ 收口 → Artifact 导出"人物诊断报告 v1" └─ 关闭对话 对话 C:综合改写方案 └─ 第一句:"我会贴两份诊断报告,你综合后给我改写方案" └─ 贴入 A + B 两份 Artifact └─ "三个改写优先级:必改 / 可改 / 不改" └─ 收口 → Artifact 导出"改写方案 v1"
为什么这样比一个对话强
• A 节点 Claude 状态新鲜,诊断锐利
• B 节点 Claude 不被 A 的诊断细节拖累,只看结论
• C 节点 Claude 对两份诊断"等距离",综合判断更平衡
并行式接力(多角度同步)
同一份稿子,开三个独立对话: 对话 A:戴医生帽诊断 → Artifact A 对话 B:戴观众帽诊断 → Artifact B 对话 C:戴投资人帽诊断 → Artifact C 最后开对话 D: - 贴入三份 Artifact - "综合三个视角,告诉我: 1. 三方都同意的问题是什么(必改) 2. 只有一方提出的问题是什么(看情况) 3. 三方互相矛盾的问题是什么(需我决策)"
注意:每个节点结束必发收口咒语。Artifact 用清晰版本号命名。节点之间转换时第一句话写明"上一节点产出 + 本节点任务",别让 Claude 自己猜。

P1
Skill 在 Cowork 跑 — 两端共用方法论
必学
›
是什么
Skill 不是 Chat 专属。把 Skill 放进 Cowork 工作区,Folder Instructions 引用,Cowork 执行任务时按同一套方法论工作。
为什么
• Chat 想方法,Cowork 执行,两端共用一份方法论 = 思维一致性
• 否则 Chat 给你的诊断框架,Cowork 出报告时用的是另一套
• 跨项目同样有效——新项目复用同一个 Skill
怎么做
1. Skill 文件夹复制到 /Cowork工作区/项目名/00_工作模板/skills/[skill-name].skill/
2. Folder Instructions 加引用段(见下)
3. 启动协议增加一步:"自检阶段需复述 Skill 是否识别成功"
Folder Instructions 追加段
# 方法论引用 本项目使用 /00_工作模板/skills/[skill-name].skill/ 执行任何分析、诊断、生成任务时: 1. 优先调用 Skill 的方法论 2. Skill 里有的流程,严格按 Skill 走 3. Skill 没覆盖的场景,先问我用什么方法,不要自己编 4. 产出文件末尾标注"基于 [skill-name] 方法论" # Skill 自检要求 每次启动协议的"自检"环节必须: - 列出识别到的 Skill 名称 - 复述 Skill 的核心流程(不超过 5 行) - 报告 Skill 是否完整可用(子文件是否齐全)
注意:Skill 路径不要随便挪,挪了 Folder Instructions 同步改。大型 Skill 在 Cowork 里跑比 Chat 慢(要读多个文件)。一个项目最多挂 1-2 个核心 Skill。

P2
多连接器联动 — Cowork 真正的威力
›
是什么
Cowork 同时连多个外部服务(Drive、Notion、Slack、本地文件、浏览器),任务可以跨工具串起来。
为什么
• 单连接器 = Cowork 当个文件助手用
• 多连接器 = Cowork 当数字助理用
• 你的素材本来就分散在各处(Drive 拍摄日志、Notion 人物卡、本地剪辑稿)
常用连接器配置(纪录片场景)
• Local Filesystem — 本地稿件、剪辑工程文件、素材
• Google Drive — 团队共享、拍摄日志、采访录音
• Notion — 人物卡、场景表、调研笔记
• Chrome 浏览器 — 在线参考、参考片信息
• Slack — 团队沟通(找特定话题、找谁说过什么)
跨工具任务模板 · 人物动机审视
跨工具人物动机审视 请按以下步骤,跨工具收集信息后做综合分析: 第一步:收集 - 从 Notion 读取「[人物名]」人物卡的完整内容 - 从 Drive 的「拍摄日志」文件夹找到这个人物相关的拍摄日志 - 从本地 02_当前稿件 读取最新粗剪稿, 定位这个人物的所有出场段落 第二步:对照 将三方信息对照: - 人物卡里设定的核心动机 - 拍摄日志里现场捕捉到的真实状态 - 粗剪稿里实际剪进去的段落 第三步:判断 - 设定动机 vs 现场真实:有没有偏差? - 现场真实 vs 剪辑呈现:有没有信息丢失? - 三场出场是不是一条递进线? 第四步:产出 存到 04_Claude产出/ 文件名:人物动机审视_[人物名]_[日期].md 执行前先列计划。
注意:第一次每个连接器都要授权。连接器越多 Cowork 越慢(每次任务要扫多源)。涉及 Slack/Notion 时注意隐私,不要让它读不该读的频道。不需要的连接器及时断开。
P3
任务模板化 — 把反复做的事封装成"配方"
必学
›
是什么
反复做的事(整理素材、归档粗剪、做镜头清单)写成 .md 任务模板,存进 /00_工作模板/。下次一句话调用。
为什么
• 每次重新组织语言费时间还容易漏
• 模板化的任务执行更稳定
• 可以不断迭代模板,把踩过的坑写进去
模板组织结构
/00_工作模板/ /tasks/ ← 任务模板 粗剪后镜头清单.md 周度素材归档.md 采访素材切片.md 人物动机审视.md /spells/ ← 启动咒语备份 启动协议.md 项目复位.md /skills/ ← 方法论 [skill-name].skill/
任务模板示例 · 粗剪后镜头清单.md
# 任务:粗剪后镜头清单生成 ## 触发 当用户说"按 粗剪后镜头清单 流程做"时执行。 ## 输入 - 读取 02_当前稿件/ 下最新粗剪剧本(按修改时间) - 自动识别版本号 ## 流程 1. 通读全文,识别所有场次划分 2. 每个场次内列出所有镜头描述 3. 每个镜头标注: - 镜号(场次-序号,如 S03-04) - 一句话内容描述 - 必要性等级(A=核心 / B=辅助 / C=可删) - 估计时长(秒) ## 输出 - 文件名:镜头清单_[剧本版本号]_[日期].md - 位置:04_Claude产出/ - 格式:Markdown 表格 - 末尾追加"删减建议":可删 C 级镜头总时长 / 占比 ## 异常处理 - 找不到粗剪剧本 → 不要继续,问我位置 - 粗剪超过 200 场 → 先问我是不是要分批做 - 镜号格式不一致 → 报告给我,不要自动统一 ## 完成汇报 - 总场次数 / 总镜头数 / A 级占比 / 可删时长 - 一行内
调用方式
一句话:"按 粗剪后镜头清单 流程做"
Cowork 自动找到模板,按流程走。
注意:模板里把"边界条件"写清楚(找不到文件怎么办、超过多少需要确认)。第一次执行先盯着,不对就改模板。每次执行后回看一次,有改进点立刻更新。
P4
多文件夹协作 — 一会话访问多工作区
›
是什么
Cowork 一个会话可以授权访问多个文件夹,不局限于一个项目目录。
为什么
• 跨项目复用资料(参考片库、模板库、共享素材)
• 不用每个项目复制一份重资料
• 团队协作场景:个人项目 + 团队共享区
推荐三件套配置

P5
Dispatch 外景工作流 — 你这种导演的杀手锏
立刻能用
›
是什么
Dispatch 让你从手机派任务给桌面 Cowork,任务在桌面后台跑,你在外面继续干别的。
为什么
• 拍摄现场 / 出差 / 通勤时间不浪费
• 整理类、归档类任务不需要你盯着
• 回到电脑前打开看产出就行
完整工作流(以拍摄日为例)
早晨出门前(桌面): 1. 确认桌面 Cowork 应用已打开 2. 设置电脑不睡眠 / 不熄屏 (Mac: Caffeine / Win: 电源设置) 3. 充电插好 4. 测试 Dispatch 配对正常 拍摄现场(手机): - 上午拍完一场: "刚拍完 S05,现场录音上传到 Drive 的 [路径], 把对白切片整理出来" - 中午吃饭时间: "读今天上午所有上传的视频文件名, 生成今日上午素材清单" - 收工: "整理今天全部素材按场次分类, 生成今日素材日报,存到 04_Claude产出/" 回酒店(桌面): - 打开电脑就看到全部产出 - 检查 / 复核 / 调整 - 第二天的拍摄计划基于今天的整理来定
适合 Dispatch 的任务类型
✅ 文件整理、归档、重命名
✅ 素材清单、日报生成
✅ 长篇资料预读 / 摘要
✅ 跨工具数据汇总
❌ 需要即时反馈的诊断/对谈(留给 Chat)
❌ 不可逆的删除/覆盖(必须当面确认)
硬限制:桌面应用必须保持打开 + 不睡眠,合上盖子任务死。早期版本约 50% 一次成功率,重要任务回家后复核。涉及不可逆操作的任务不要 Dispatch,留到回家做。
优先级建议
不用一次全上。按这个顺序最值得:
① A1 + P1(Skill 体系两端共用)
沉淀效应最大,半年后回看会感激自己。

② P3(任务模板化)
重复劳动直接砍半。每周做的事先封 3-5 个模板。

③ A3 多模态 + P5 Dispatch
你的工作场景立刻能用上。下次拍摄就试。

④ A4 触发词深玩法 + A5 接力工作流
单次产出质量上一个台阶。

⑤ A2 Style + P2 多连接器 + P4 多文件夹
长期使用的隐形优化。

避坑清单
8 个最常见的错误用法,直接对照纠正。
✗在 Chat 里挤牙膏式给信息
✓一次性贴齐所有上下文
✗一个对话里塞 5 个不相关问题
✓一轮聚焦一件事
✗Chat 对话从不收口
✓每次结束都强制收口,把总结存档
✗让 Cowork 访问整个硬盘
✓只授权专门的工作文件夹
✗Cowork 任务说一句模糊指令
✓五要素齐全 + 先要计划
✗Cowork 产出直接当最终稿
✓必须人工打开复核
✗把 Chat 和 Cowork 当同一种工具用
✓思考用 Chat,执行用 Cowork
✗全程用同一个模型
✓深度任务 Opus,迭代任务 Sonnet
头号大坑:把 Chat 当 Cowork 用——指望发一句话就拿到完整产出。Chat 的真正价值是多轮对谈中思维的递进。
