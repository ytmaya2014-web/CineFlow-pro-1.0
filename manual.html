Director-Writer Collaboration System

导演型
写作协作系统

一套面向纪录片与宣传片创作的 Skills 体系。从创意发散到结构搭建， 从声部编织到稿件迭代，让每一个环节都有其所、各归其位。

CHAPTER / 01

这是什么

一套让你的影视文案创作从"靠直觉"变成"有纪律"的协作系统。

这不是一个单一的工具，而是一组各司其职的 Skills， 通过标准化的交接协议串联成完整工作流，覆盖纪录片与宣传片创作的每个关键环节。

解决什么问题

一个合格的影视文案从业者，头脑里其实有一套潜在的工作流—— 从听到命题、发散创意、搭骨架、选素材、写稿、诊断、改稿、定稿。 但它通常是隐性的，藏在经验里，难以复现，也难以教给他人。

这套系统做三件事：

  • 显性化——把隐性的创作流程拆成可看见的阶段与动作
  • 标准化——让每个阶段的产出能被下一阶段直接消费，不需要返工翻译
  • 纪律化——给每个 skill 明确的入场与退场条件，避免工具之间互相干扰

为谁而设计

适合 · FOR
  • 纪录片、宣传片、品牌片导演与策划
  • 有一定经验、但希望把方法论沉淀下来的创作者
  • 带团队的人，想把自己的经验传递给新人
  • 处理多版本、多声部、多轮迭代的复杂项目
不适合 · NOT FOR
  • 短视频的 30 秒脚本（杀鸡用牛刀）
  • 纯文本的公众号推文（这是另一个体系）
  • 需要 AI 替你做所有判断的人（这套系统强调导演的判断力优于工具）
  • 希望一键出稿的人（这不是模板生成器）
核心信念
圆桌是脚手架，不是建筑本身。最终的判断来自导演的综合素养，不来自工具。 当工具的结论和直觉相反时，先问直觉从哪里来。

整体流程图

一个典型项目，从命题到定稿的完整路径。

—— Project Flow · 项目主流程 ——

流程不是机械的。不同项目走的路径会有所不同——

  • 全新复杂项目：完整走过所有阶段
  • 只有改稿任务：从"三顶帽子"进入，直接处理修改
  • 短片或单声部：跳过 voice-weaving，缩短链路
  • 纯拍摄执行：从成稿后直接进入执行转化
关于 style-learner
它不是流程里的一个"阶段"，而是一个背景进程。 在你使用其他任何 skill 时，它都在默默观察你的风格偏好，并在跨对话时保留这些记忆。 你无需主动调用它——它会自己出现。

Skills 详细介绍

每一个 skill 的定位、触发条件、核心方法、以及它和其他 skill 如何交接。

S-01 · round-table｜六人圆桌创意系统

一句话定位
当你拿到一个命题但还没形成判断时，用六种认知框架碰撞出创意。

什么时候启用

  • 命题刚到手，还不知道怎么切入
  • 有几个可能的方向，但拿不准哪个最锋利
  • 项目涉及多方利益（甲方 / 受众 / 行业 / 合规），需要多视角碰撞
  • 团队讨论陷入僵局，需要结构化突破

什么时候不启用

  • 命题已经很明确，洞察已经成形 → 直接去 doc-methodology 建框架
  • 纯执行任务（比如写一段 30 秒口播）
  • 时间极度紧张（一次完整圆桌要 30-60 分钟）

核心角色

角色信息偏好惯用武器
消费者代言人只信具体的人的具体故事你说了半天，跟我有什么关系？
品牌叙事官只信情感内核和可见时刻态度必须可见，藏在一个具体的时刻里
直觉创意人只信自己身体的第一反应感觉对就是对
沉默者全程不发言，只听最后说两句，往往最真实
+ 两把动态椅子（按项目类型替换：商业 / 政务 / 文旅）

五幕结构

  1. 第零幕 · 命题校准：每个角色复述命题，如有分歧说明命题本身需要澄清
  2. 第一幕 · 独白期：六个角色独立发言，每人必须用"感知描述"表达立场
  3. 第二幕 · 碰撞期：沿对立链定向攻击，中途强制触发导演表态
  4. 第三幕 · 重建期：每个角色承认对手有一点是对的，提炼"被迫共识"
  5. 第四幕 · 命题尸检：如果项目失败了，最可能的死因是什么？

输出

  • 最终洞察：命题转化成了什么
  • 方案纵深：甲方一年后才会看见的那一层
  • 最小验证：24 小时内能做的最小动作
  • 交接包 → 下一站通常是 doc-methodology 或 director-writer
关键特色：缺席者席位机制

在开圆桌之前，导演必须回答：这个项目里最关键的缺席者是谁？通常是甲方董事长或最终决策者。

缺席者需要三个时间切面同时在场——

  • 现在：他今天签字同意的判断标准
  • 半年后：根据他的行业、政策、组织压力，他可能的变化方向
  • 交付时：成果真正出现在他面前那一刻，他会用什么标准评判

核心原则：真正的前瞻性不是预测领导怎么变，而是做出在任何合理变化下都能站住的方向。

S-02 · doc-methodology｜纪录片方法论系统

一句话定位
融合好莱坞叙事大师的方法论，帮你在思考阶段就建立科学的叙事框架。

两个入口

入口 A · 从零思考

七层提问法

  1. 故事问题（Story Question）
  2. 控制理念（Controlling Idea）
  3. 核心人物与价值转变
  4. 叙事模式选择
  5. 三幕结构适配
  6. 情感弧线设计
  7. 叙事线索检验
入口 B · 已有稿件

三顶帽子

  • 🎩 医生帽 · 冷静挑刺
  • 🎬 编剧帽 · 建设性建议
  • 👀 观众帽 · 模拟观众反应

同一份稿子，换帽子就换大脑。

三顶帽子的工作方式

帽子人格只做不做
🎩 医生帽 冷静精准，不留情面 挑刺、诊断病灶 不给建议、不给药方
🎬 编剧帽 热情、充满可能性 给方向、给方案、给灵感 不只批评、不停在问题层面
👀 观众帽 普通人、只有直觉 说感受、问问题 不说术语、不给答案

关键原则

  • 每层提问一次只问 1-2 个，等回答后再推进
  • 三顶帽子切换时要"重置视角"——假装之前没看过稿子
  • 三顶帽子结论冲突时——那正是稿件的核心张力所在，最值得关注
  • 本 skill 不出稿，只出思路；出稿请交给 director-writer
工具箱：五个检验工具
  • 一句话测试：超过 30 字 = 没想清楚；需要"然后" = 流水账
  • 酒吧测试：在酒吧向陌生人讲——他追问 = 有效；他走神 = 重来
  • 倒推法：从控制理念倒推证据顺序，对比稿子实际顺序
  • 减法测试：逐段问"删掉会不会影响后面"——不会就删
  • 沉默替代法：删掉旁白，看画面能不能讲 60% 的故事

S-03 · voice-weaving｜多声部叙事编织系统 [ NEW ]

一句话定位
专门解决"两条或以上叙事线"的设计问题。定位格、排节奏、做避撞。

为什么要专门做这个 skill

纪录片项目里最难的一种情况：你同时想要"意义层"和"故事层"、"冷视角"和"热视角"、"旁观者"和"当事人"—— 两条线各自都能说明问题，但合在一起就打架。这不是文字问题，是声部设计问题。

以前的处理方式是靠直觉"硬整合"，耗时耗神且结果不稳定。 这个 skill 把声部设计变成可拆解、可复现的工序。

核心概念 · 声部的六个位格属性

属性选项示例（心域线）示例（故事线）
距离远 / 中 / 近远近
温度冷 / 温 / 热冷重
身份旁观者 / 当事人 / 见证者 / 上帝视角旁观者当事人
时间过去 / 当下 / 未来 / 永恒永恒当下
任务提问 / 陈述 / 解释 / 追问 / 收束追问+收束陈述+见证
语言质地格言 / 叙事 / 口语 / 诗化 / 数据化诗化+格言叙事+口语

每一条声部，六个属性都要明确，一项都不能模糊。否则声部会在成稿过程中失格。

六种常见交叉模式

结构类
  • 交替式：A→B→A→B 严格轮流
  • 主辅式：A 主 B 辅，B 在关键处出现
  • 包裹式：B 开场→A 主体→B 收尾
动态类
  • 递进式：重心从 A 向 B 转移
  • 对位式：A 说一事，B 从反方向呼应
  • 消长式：A 渐退，B 渐进（或反之）

避撞检查三项

  • 撞车检查：两条声部在同一场景里说了同一件事？互相翻译？抢位？
  • 失格检查：心域线有没有下场当旁白？故事线有没有突然说格言？
  • 稀释检查：两条都很满的场景是否反而削弱情绪？该让一条让位
合并双版稿的五步流程

当你带着"两个独立写好的稿子"来时——就像《心域的证言》最初的情况——不要直接粘贴，按下面五步：

  1. 提纯每版稿子的 DNA（最好的 5 句、核心任务、六属性）
  2. 诊断冲突点（重复 / 对位 / 独占）
  3. 重新分配任务（谁提问谁陈述谁收束）
  4. 重排出场节奏（按幕列表）
  5. 输出交接包给 director-writer 成稿

S-04 · director-writer｜导演型写作系统

一句话定位
整个体系的中心枢纽。绝大部分写作任务最终在这里完成。

五种工作模式

模式触发信号产出
🏗️ 结构搭建 圆桌/方法论刚做完，要定三幕 三幕骨架图 + 情感弧线 + 声部需求判断
🔍 稿件诊断 "有没有问题？哪里要改？" 结构化诊断报告 + 修改任务清单
✂️ 母稿拆版 政府版/客户版/业主版/长短版 共用骨架 + 各版本差异 + 成稿
✍️ 成稿生成 "帮我写完整稿" 完整解说词 / 脚本 / .docx 文档
🎬 执行转化 定稿后要拍摄 拍摄版 + 采访提纲 + 画面对应表

模式自动判断

你不需要手动指定模式。director-writer 会根据你的输入自动判断：

  • 只有创意洞察 → 进结构搭建
  • 有初稿说表达不满意 → 进稿件诊断
  • 有成稿要多版本 → 进母稿拆版
  • 要出完整新稿 → 进成稿生成
  • 有文字稿要落地拍摄 → 进执行转化

每次任务前的六项判断

  1. 行业语境（房地产 / 政务 / 文旅 / 公益 …）
  2. 片型（宣传片 / 纪录片 / 汇报片 / 短视频 …）
  3. 受众（决定语调，语调决定一切）
  4. 项目阶段（资料 / 结构 / 初稿 / 成稿 / 执行）
  5. 最强说服来源（人物 / 数据 / 过程 / 场景）
  6. 当前最需要推进的事
受众语调速查表
受众语调禁忌
政府 / 主管部门庄重、宏观、排比有力过度煽情、个人视角
大众消费者温情、生活化、从小处切入术语堆砌、数据轰炸
企业客户 B 端专业、理性、价值导向空喊口号
年轻受众 / 社交媒体直接、节奏快、强钩子说教、冗长开场
行业专家克制、准确、以事实说话过度修辞
公益 / 社会议题真实、克制、留白说教、过度悲情

S-05 · iteration-manager｜稿件迭代管理系统 [ NEW ]

一句话定位
给多轮迭代装上仪表盘。一稿一焦点，有纪律地前进，有勇气地封稿。

解决什么问题

没有迭代纪律的灾难
  • 永远改不完：每次发现新问题，从不收口
  • 越改越散：改了 A 破坏了 B，修了 B 之前的 C 又回退
  • 改丢初心：定稿时连最初要表达的都模糊了
有纪律的迭代
  • 一稿一焦点：最多 3 个问题，通常 1-2 个
  • 保留清单先于修改清单
  • 推迟的权利：不是所有问题都要立刻解决
  • 留"化石"：每稿改动都标注来源
  • 封稿的勇气：到点就停

三种使用场景

  1. 开始新一稿：定本稿要解决什么、保留什么、推迟什么
  2. 多稿之间的 diff 追踪：每稿解决了什么、副作用是什么
  3. 封稿判断：走"封稿检查表"决定能不能定稿

封稿检查表（六层 + 三顶帽子回顾）

  • 母题层：一句话能说清要证明什么吗？比 v1 时更清晰还是更模糊？
  • 结构层：三幕边界清楚吗？有没有中点翻转？情感曲线在你想要的位置吗？
  • 语言层：有没有"AI 味"？有没有甲方腔？
  • 声部层：位格有没有混？有没有互相翻译？
  • 可拍摄层：每一场景都能说清"谁说、拍什么"吗？
  • 留白层：有没有给观众消化的空间？结尾让人沉默还是鼓掌？
关于封稿
如果你已经改到 v7 以上还在犹豫——这稿可能已经足够好了。继续改的边际收益在递减。 如果你在等一个"更好的版本"，它可能永远不会来。建议再精修一轮，封稿，把精力留给下一个项目。

S-06 · style-learner｜写作风格学习系统

一句话定位
一个有天赋的写作学徒。跨对话记忆你的语言偏好，让你无需每次重复"我喜欢什么"。

工作原理

style-learner 不是一个主动调用的工具，它是一个背景进程。 在你和任何一个写作类 skill 交互时，它都在默默观察——

  • 你对第一稿做了哪些修改（替换了哪些词，重组了哪些句）
  • 你明确表达了哪些偏好（"不要用这个词"、"太长了"、"语气再硬一点"）
  • 你在多个版本中选了哪个
  • 你提供的原始素材的语言倾向

这些观察会被整理成一份 Style Profile（风格档案）。 下次新对话时，只要把档案粘贴进来，Claude 就能直接按你的标准工作，无需从零磨合。

三种工作模式

模式触发行为
A · 首次观察 没有档案 静默观察，攒够 3 个信号后主动提议建档
B · 档案注入 用户粘贴档案 内化偏好，按标准工作
C · 档案更新 用户说"更新"或产生新信号 整理信号、版本+1、输出新档案
多声部项目的特别字段

对于多声部纪录片项目，style-learner 的档案中会单独记录每条声部的语言特征——

比如《心域的证言》的风格档案里会明确：

  • 心域线：位格（远/冷/旁观/永恒/追问+收束/诗化+格言）、标志句式、禁区
  • 故事线：位格（近/重/当事/当下/陈述+见证/叙事+口语）、标志句式、禁区

这样下次再做类似多声部项目，可以直接把档案作为参考，而不是重新推敲。

归档说明 · pastor / pastor-writing

为什么归档
PASTOR 是销售文案框架（Problem → Amplify → Story → Testimony → Offer → Response）， 强行套在纪录片/宣传片上会产生以下问题——
  • 被迫设置"问题→放大→解决"逻辑，但纪录片未必有这个结构
  • 强制的 Offer + Response 往"行动号召"走，结尾变口号
  • 沉思型纪录片（如《心域的证言》）完全不需要 PASTOR 逻辑

被谁取代：director-writer 取代 pastor（影视 PASTOR）。 pastor-writing（销售长文案版本）作为单独工具保留，但不进入纪录片工作链。

Handoff 协议

Skills 之间的交接标准。让工具链真正连上手，不要让用户当搬运工。

为什么需要这个协议

每个 skill 做的事情不同，但它们要协同完成一个完整的项目。 圆桌产出的"洞察"，要能被结构搭建 skill 直接用； 诊断产出的"问题清单"，要能被成稿 skill 直接消费—— 不能每次都要用户手动复述、手动翻译、手动整理。

核心原则
每个 skill 的输出 = 下一个 skill 的输入。
不要让用户在中间当搬运工。
不要让 Claude 在下一个 skill 里说"请告诉我上一步的结果是什么"。

标准交接包格式

任何 skill 在结束工作时，如果判断接下来会转入另一个 skill，必须以以下格式提供交接信息——

From: [当前skill名]
To:   [推荐的下一个skill名]
═══════════════════════════════════════

【项目基础信息】
- 母题：（一句话）
- 片型：
- 受众：
- 阶段：

【本阶段的核心产出】
- 关键结论 1 ...
- 关键结论 2 ...
- 关键结论 3 ...

【给下一个 skill 的任务指令】
1. ...
2. ...
3. ...

【需要携带到下一阶段的约束】
- 不能改变的：
- 需要保护的：
- 明确禁止的：

【待补充信息】
- [ ] 项目 1
- [ ] 项目 2

路径 A · 全新项目从创意到成稿

round-table
    ↓ (交接包：三重结论 + 被迫共识)
doc-methodology (七层提问)
    ↓ (交接包：叙事框架 + 声部需求判断)
voice-weaving (如需多声部)
    ↓ (交接包：声部设计方案 + 节奏表)
director-writer (成稿模式)

路径 B · 诊断已有稿件

doc-methodology (三顶帽子)
    ↓ (交接包：致命问题清单 + 可保留清单)
director-writer (诊断模式 或 成稿模式)
    ↓ (如进入迭代循环)
iteration-manager (管理本稿要改什么)

路径 C · 成稿到拍摄

director-writer (成稿) → iteration-manager (封稿判断)
    ↓ (确认封稿)
director-writer (执行转化模式)
    ↓

使用原则

  • 能用结构化字段就不要用散文——Claude 和下次对话都能稳定解析
  • 字段留空也要保留——写"（暂无）"，让用户知道这里要考虑但暂时没答案
  • 用户可以在交接包里直接修改——交接包不是只读的，用户改了什么下一个 skill 必须尊重
  • 每次 handoff 之后旧 skill 就退场——避免两个 skill 同时在线互相干扰

CHAPTER / 06

实战案例 · 心域的证言

一个真实的纪录片项目，从零到定稿的完整流程。看每一步用了什么 skill、产出了什么、关键决策在哪里。

项目背景

项目《心域的证言》时代首府纪录片 · 业主版
甲方浩宇集团（房地产开发商）
受众已购房业主 + 潜在高端客户
片长约 18-22 分钟（完整文学脚本）
核心挑战甲方既要"精神高度"又要"建造真实"；市场上地产纪录片严重同质化
迭代次数7-9 稿（到 v7.2 封稿）

流程复盘

01
STAGE 01 · 创意阶段 · round-table

开圆桌，碰撞出创意方向

项目命题到手后，先开一场六人圆桌。六种认知框架对命题进行攻击与重建——

产出的最终洞察：这不应是一部介绍项目的片，而是一部关于"不将就"的纪录片—— 建造者的不将就与居住者的不将就，是同一把尺子。

方案纵深：业主看的时候理解"他们很认真"，一年后住进去、某天摸到阳台边沿的圆弧时， 才会真正懂这部片子在说什么。

沉默者的两句话： "真正重要的是'看不见的较真'"；"所有人都在说产品力，没人敢说'配得上'。"

02
STAGE 01 · 创意阶段 · doc-methodology

用七层提问建立叙事框架

带着圆桌的洞察进入 doc-methodology——

  • 故事问题：什么样的房子，配得上你？谁决定了这个"配得上"？
  • 控制理念：真正的品质不在外立面的恢弘里，在看不见的较真里——因为建造者和居住者之间，最深的默契是"你看不见的地方，我也没有将就"。
  • 叙事模式：混合型——解说主导 + 观察型 + 诗意型
  • 情感弧线：开场低温沉重 → 第二幕压抑（脚手架、拆重装、除夕夜）→ 第三幕舒缓（松石园林）→ 结尾极静（空楼走楼）
03
STAGE 02 · 结构阶段 · director-writer 结构搭建模式

搭三幕骨架

  • 序 · 两个声音：王总背影 + 那个问题"住在这里的人回过头来会说什么"
  • 第一幕 · 那份不将就：拿地、看项目、看石材、脚手架、图纸前的刘树海
  • 第二幕 · 压力往下走：四层递进结构（看得见的 → 看不见的 → 最小的 → 人的代价）
  • 第三幕 · 压力的形状：松树、樱花、园林、白手套
  • 结尾 · 走楼：所有人退场，摄影机独行

关键判断：这是多声部项目——转 voice-weaving。

04
STAGE 02 · 结构阶段 · voice-weaving

设计双声部：心域线 + 故事线

属性心域线故事线
距离远近
温度冷重
身份旁观者当事人
时间永恒当下
任务追问 + 收束陈述 + 见证
语言质地诗化 + 格言叙事 + 口语

交叉模式：主辅式 + 对位式。故事线是主干（约 75% 篇幅），心域线在每幕开头/结尾/层次转换处出现。

避撞原则：心域线绝不下场讲具体事实；故事线绝不升格说道理；两条线不在同一场景里说同一件事。

05
STAGE 02.5 · 素材阶段 · 人工 + Coworker

有目的地调研

带着清晰的结构和声部需求去采集素材——不是漫无目的的采集，是针对每一场景的明确补位：

  • 补采刘树海的同期声（铝板色差 f316、腊月 23:59:59 的故事）
  • 补采李强林（"服气了"这个原话）
  • 补采园林设计师的宋式美学阐述
  • 拍摄执行清单里的档案照片与现场特写
06
STAGE 03 · 成稿阶段 · director-writer 成稿模式

按骨架与声部方案成稿

这是最具体也最长的阶段。每写完一场景，自检三项——

  • 每句旁白的声部归属明确吗？
  • 两条声部本场景的出场比例和节奏表一致吗？
  • 有没有互相翻译？
好的材料不抢戏。它退在后面，让住在里面的人往前站。 —— 心域线 · 千里看石材场景
服气了。 —— 故事线 · 李强林同期声（不追问服什么气，让它停在那里）
07
STAGE 03 · 迭代阶段 · 三顶帽子 × iteration-manager

7-9 稿的有纪律迭代

每一稿明确本稿要解决的问题，用三顶帽子诊断：

版本本稿要解决保留推迟
v1搭骨架，双线各自独立—语言打磨
v3心域线与故事线的整合骨架语言、节奏
v5心域线降温、去"AI 味"格言整合后结构层次入口的过渡
v7四层递进的过渡设计语言个别场景的采访留白
v7.2暗线勾连五处锚点（序章、第一幕末、第二幕中段、第三幕、结尾）整体—（定稿）

稿件里的 【v7.2】 标记是迭代过程的化石——每一处都对应一个修改目标，封稿前统一清理。

08
STAGE 04 · 封稿 · iteration-manager

封稿检查表通过，定稿

v7.2 封稿检查六层全过——

  • ✓ 母题清晰可一句话说出
  • ✓ 三幕边界明确，有中点翻转（场景九拆重装）
  • ✓ 语言克制，无 AI 味格言
  • ✓ 双声部位格稳定，无互相翻译
  • ✓ 每一场景可拍摄，有明确画面指引
  • ✓ 结尾是"让人沉默"，不是"让人鼓掌"
09
STAGE 04 · 交付 · director-writer 执行转化模式

转拍摄版、采访提纲、画面对应表

文学脚本定稿后，转入执行——

  • 每场景标注：谁说（旁白 / 同期声 / 字幕）
  • 每场景标注：拍什么（具体机位、焦段、光线、运动）
  • 补采访提纲（贾孝政等待访人员的针对性提问）
  • 档案照片清单（需从现有素材库调取的部分）
复盘·关键决策
整个项目最关键的决策是在 voice-weaving 阶段作出的——把两条独立的稿子"合并"这个念头。 如果没有 voice-weaving 这样的工具化工序，这一步只能靠直觉硬整合，结果可能是两条线互相稀释。 有了位格分配 + 节奏表 + 避撞检查，两条线才能既各自为政又彼此呼应。

CHAPTER / 07

常见场景手册

五种最常见的项目场景，每一种对应一条最短路径。

场景 A · 全新项目，从零开始

典型语境：甲方刚签了意向书，你拿到命题但还没动笔。

推荐路径：

  1. 建项目记忆文档（母题、受众、约束、缺席者）
  2. 开 round-table 圆桌 → 产出三重结论
  3. 转 doc-methodology 七层提问 → 产出叙事框架
  4. 转 director-writer 结构搭建模式 → 产出三幕骨架
  5. 如多声部 → voice-weaving 设计声部
  6. 有目的调研素材（人工 + Coworker）
  7. 转 director-writer 成稿模式 → 产出 v1
  8. 进入迭代循环（诊断 × 修改 × 迭代管理）

时间估算：结构+骨架约 3-5 天；成稿约 1-2 周；迭代约 2-4 周；总周期 1-2 个月。

场景 B · 已有初稿，想诊断

典型语境：同事写了一稿给你看，或者你自己的稿子感觉哪里不对但说不清。

推荐路径：

  1. 直接进 doc-methodology 三顶帽子（建议先戴医生帽）
  2. 拿到诊断交接包后，转 director-writer 诊断模式或成稿模式
  3. 如果要改 → 先走 iteration-manager 定本稿要解决什么
  4. 按任务清单改稿 → 再次诊断

关键点：不要跳过三顶帽子直接改。你的直觉"哪里不对"需要被翻译成具体问题才能解决。

场景 C · 要做两条叙事线

典型语境：甲方要"高度"又要"温度"；或你已经写了两版稿子想合并。

推荐路径：

  1. 先去 director-writer 结构搭建模式确认真的需要双声部
  2. 如确认双声部 → voice-weaving 做位格设计
  3. 制定避撞原则和节奏表
  4. 转 director-writer 成稿模式按方案写
  5. 每个场景写完自检：声部归属、节奏、避撞
警告
不要在不需要双声部的项目里硬加双声部。8 分钟以下的短片、纯人物故事、资讯类短视频， 单声部几乎永远更好。
场景 D · 稿子改了好几稿停不下来

典型语境：v5、v6、v7 一直在改，但不知道还要改到什么时候。

推荐路径：

  1. 立刻开 iteration-manager
  2. 做 diff 追踪表：每一稿解决了什么、副作用是什么
  3. 如果发现"同一问题反复出现"或"改了又改回去"——警示信号
  4. 走封稿检查表六层
  5. 按结果决定：封稿 / 再精修一轮 / 结构性重改

关键判断：如果你在 v7 及以上还在犹豫，大概率这稿已经足够好了。继续改的边际收益在递减。

场景 E · 稿子定了要拍摄

典型语境：文学脚本封稿，进入拍摄筹备。

推荐路径：

  1. 转 director-writer 执行转化模式
  2. 按场景逐段拆分：说什么、谁说、拍什么、用什么同期声、需要哪些资料画面
  3. 对需要访谈的人物，补出采访提纲
  4. 对现有素材库，做画面与文案的对应表
  5. 标注"必须补拍"和"档案可用"的分类

CHAPTER / 08

培训指南

如果你在给团队或新人讲这套系统，建议按这个顺序和重点来。

三个必须先讲的核心理念

理念 ①
工具是脚手架，不是建筑本身。
Skills 不替代导演的判断力。它们帮你把隐性的经验显性化，把直觉的判断纪律化。 最终的决定权永远在你手里。
理念 ②
每个 skill 有严格的职责边界。
doc-methodology 不出稿，只出思路。director-writer 不发散创意，只做判断与成稿。 voice-weaving 不管结构，只管声部。越是边界清晰，协作越顺。
理念 ③
Handoff 是体系的灵魂。
一个好的交接包胜过十次重复沟通。每次 skill 切换时，都要检查交接包的完整性—— 这是整个系统能否流畅运转的命脉。

新人学习路径

阶段先学什么推荐 Skills
第 1 周 · 理解 整体流程图 + 层次划分；熟悉术语 只需要读本手册第 1-3 章
第 2-3 周 · 诊断练习 用三顶帽子诊断历史稿件 doc-methodology
第 4-6 周 · 结构练习 用七层提问从零做一个短片 doc-methodology + director-writer 结构搭建
第 2 月 · 成稿练习 完整跑一个真实项目 全流程，跳过 voice-weaving
第 3 月 · 双声部挑战 加入 voice-weaving，做复杂项目 全流程 + 迭代管理

常见误用

  • 跳过圆桌直接写稿——创意不扎实，后期改再多也回不去
  • 在 doc-methodology 里要求出稿——走错门，应该转 director-writer
  • 一稿要解决一堆问题——违反"一稿一焦点"原则，结果改到最后一团乱麻
  • 不做声部设计直接写双线——两条线互相稀释，白花力气
  • 没人封稿，反复修改——没有 iteration-manager 的纪律，永远改不完
  • 把 pastor 用在纪录片上——框架不对，写什么都像广告

FAQ

Q：我的项目很简单，需要完整走完所有 skills 吗？

不需要。这套系统是"可伸缩"的——

  • 30 秒短视频：直接 director-writer 成稿模式即可
  • 5 分钟品牌片：七层提问 + 结构搭建 + 成稿
  • 10 分钟宣传片：加入三顶帽子诊断
  • 纪录片 / 复杂项目：完整流程
Q：每个 skill 都要手动激活吗？会不会很麻烦？

不麻烦。每个 skill 都有自动触发的关键词，你说"开圆桌"、"戴医生帽"、"搭骨架"这类话时， Claude 会自动判断该用哪个 skill。你的任务是理解这些 skill 的职责，知道什么时候该叫哪一个—— 具体的激活逻辑是 Claude 的事。

Q：Style Profile 档案怎么保存和使用？

三种方式——

  1. 对话内粘贴：新对话开头粘贴档案，Claude 即按标准工作
  2. Project Instructions：把档案放进 Claude 项目的指令里，该项目所有对话自动加载
  3. 跨账号同步：档案是纯文本，可复制到其他账号
Q：如果 Claude 做的结果和我的直觉不一样，听谁的？

听你的直觉，但先问直觉从哪里来。

Claude 的建议有可能忽略了你多年经验里沉淀的某个重要维度。 但也有可能，你的直觉只是习惯性的偏见。区分的办法是——

  • 如果你能说清楚直觉的来源（经验、案例、教训）→ 听直觉
  • 如果你只是"感觉不对"但说不出为什么 → 再想想，可能是惯性
